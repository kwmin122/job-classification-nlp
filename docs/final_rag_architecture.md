# 최종 RAG 및 역량 격차 분석 설계

Generated: 2026-05-21
Status: Final-product architecture target

## 1. 핵심 방향

최종 시스템은 사용자가 입력한 채용공고와 지원자 자료를 직접 분석한다. 이전처럼 `skill_gaps`가 이미 만들어져 있다고 가정하지 않는다.

```text
채용공고 URL/텍스트/파일
지원자 자소서/이력서/포트폴리오 텍스트/파일
        ↓
텍스트 추출
        ↓
직무 분류
        ↓
요구 역량 추출
보유 역량 추출
        ↓
역량 격차 분석
        ↓
부족 역량 query 생성
        ↓
학습자료 DB RAG 검색
        ↓
추천 자료, 로드맵, 리포트 생성
```

## 2. 입력 계층

### 채용공고 입력

- URL 입력
- 본문 텍스트 붙여넣기
- PDF/TXT 파일 업로드

URL 입력은 가능한 경우 HTML 본문을 추출한다. 채용 사이트 구조가 다르거나 차단되면 사용자에게 텍스트 붙여넣기를 요청한다.

### 지원자 자료 입력

- 자소서 텍스트
- 이력서 PDF/TXT 파일
- 포트폴리오 설명
- GitHub README 또는 프로젝트 설명 텍스트

개인정보가 포함될 수 있으므로 로컬 처리와 업로드 범위 설명이 필요하다.

## 3. 텍스트 추출 및 전처리

전처리 단계는 다음을 수행한다.

- PDF/TXT에서 텍스트 추출
- URL HTML에서 본문 후보 추출
- 불필요한 공백과 중복 줄 제거
- 문장 단위 분리
- 기술명 표기 정규화
- 너무 짧거나 의미 없는 문장 제거

DOCX/HWP는 확장 기능이다. 최종 버티컬 슬라이스에서는 PDF/TXT와 텍스트 입력을 우선한다.

## 4. 직무 분류

지원 직무군은 4개로 제한한다.

- 데이터 분석가
- AI/ML 엔지니어
- 백엔드 개발자
- 프론트엔드 개발자

모델 비교는 수업 내용 연결을 위해 유지한다.

| 모델 | 역할 |
|---|---|
| TF-IDF + SVM | baseline |
| LSTM | 순서 정보 반영 비교 |
| Text-CNN | 기술 표현 패턴 비교 |
| BERT/KoBERT | 최종 후보 |

최종 구현에서는 성능과 재현성을 기준으로 하나의 직무 분류 모델을 선택한다.

## 5. 역량 추출

채용공고에서는 `요구 역량`을 추출한다.

예시:

- Docker 기반 배포 경험
- AWS 또는 클라우드 운영 경험
- CI/CD 구축 경험

지원자 자료에서는 `보유 역량`을 추출한다.

예시:

- Spring Boot 기반 API 개발 경험
- MySQL 사용 경험
- REST API 설계 경험

추출 방식은 다음을 조합한다.

- 기술명 사전
- 직무별 핵심 역량 목록
- 문장 단위 요구사항 분류
- BERT/Sentence-BERT 기반 의미 표현
- 규칙 기반 후처리

## 6. 역량 격차 분석

각 JD 요구 역량에 대해 지원자 자료에서 의미적으로 대응되는 경험을 찾는다.

```text
JD 요구 역량:
"Docker 기반 서비스 배포 경험"

지원자 자료:
"Spring Boot API를 개발하고 로컬에서 실행했습니다."

판정:
Docker 배포 경험은 명확히 나타나지 않으므로 부족 역량으로 판단
```

gap score는 0~100으로 표현한다.

| 구간 | 의미 |
|---:|---|
| 0~39 | 낮음, 표현 보완 또는 심화 수준 |
| 40~69 | 중간, 실습 경험 보강 필요 |
| 70~100 | 높음, 핵심 경험 부족 |

gap score는 다음 요소를 조합한다.

- JD에서의 중요도
- 지원자 자료에서의 직접 언급 여부
- 의미 유사도
- 근거 문장 존재 여부
- 직무별 필수 역량 여부

## 7. RAG 검색 대상

RAG 검색 대상은 `backend/app/data/learning_resources.csv`다. 웹 전체를 실시간 검색하지 않는다.

학습자료 1행을 chunk 1개로 본다.

```text
chunk text =
job_group + skill + sub_skill + title + description + reason
```

이 프로젝트는 긴 PDF 문서 RAG가 아니라 학습자료 메타데이터 추천 RAG다. 따라서 500자 단위 문서 청킹보다 `1 resource row = 1 chunk`가 적합하다.

## 8. 임베딩 모델

최종 추천 검색은 OpenAI `text-embedding-3-small`을 기준으로 설계한다.

선택 이유:

- 한국어 설명과 영어 기술명이 섞인 자료에 대응 가능
- API 호출만으로 빠르게 embedding 생성 가능
- 80개 자료 규모에서는 별도 벡터 DB 서버 없이도 충분함
- 발표 재현성이 좋음
- 비용과 구현 복잡도가 낮음

API key가 없거나 실패하면 TF-IDF 검색으로 fallback한다. fallback은 데모 안정성을 위한 장치이지 최종 성능 주장 모델이 아니다.

## 9. 추천 점수

```text
recommend_score =
100 * (
  0.6 * semantic_similarity
+ 0.2 * skill_match
+ 0.1 * job_group_match
+ 0.1 * reliability_norm
)
```

추천 점수는 자료 검색 결과를 사용자에게 설명하기 위한 정렬 기준이다. 역량 격차 판단 점수와 혼동하면 안 된다.

## 10. 리포트 생성

리포트 생성은 새로운 판단을 하는 단계가 아니다. 이미 계산된 다음 결과를 사람이 읽기 좋게 설명한다.

- 예측 직무
- 적합도
- 요구 역량
- 보유 역량
- 부족 역량과 gap score
- 근거 문장
- 추천 자료
- 학습 로드맵

LLM을 사용할 경우에도 없는 경험이나 근거를 만들어내면 안 된다.

## 11. API 구조 목표

최종 목표 API는 다음과 같다.

### `POST /analyze`

원본 채용공고와 지원자 자료를 입력받아 전체 분석과 추천을 한 번에 수행한다.

```json
{
  "job_posting": {
    "source_type": "url",
    "url": "https://example.com/job/123",
    "text": ""
  },
  "candidate_materials": [
    {
      "source_type": "text",
      "label": "자소서",
      "text": "지원자 자기소개서 본문"
    }
  ]
}
```

### `POST /recommend`

이미 계산된 `skill_gaps`를 받아 추천만 수행하는 내부/개발용 엔드포인트로 유지할 수 있다.

## 12. 현재 구현 격차

현재 저장소는 추천 파이프라인 후반부가 먼저 구현되어 있다.

구현되어 있는 것:

- 학습자료 DB 80개
- 리소스 검색
- 추천 점수 계산
- 로드맵 생성
- 리포트 생성
- 대시보드 일부

아직 필요한 것:

- 원본 채용공고 입력 UI
- 원본 지원자 자료 입력 UI
- URL 본문 추출
- PDF/TXT 텍스트 추출
- 요구 역량 추출
- 보유 역량 추출
- gap score 직접 계산
- `/analyze` 통합 API

## 13. 발표용 설명

> 최종 시스템은 사용자가 채용공고와 본인의 자소서, 이력서, 포트폴리오 자료를 입력하면, 채용공고의 요구 역량과 지원자 자료의 보유 역량을 비교해 부족 역량과 부족 정도를 계산합니다. 이후 부족 역량을 query로 변환해 직접 구축한 한국어 학습자료 DB에서 관련 자료를 검색하고, 추천 점수에 따라 학습 로드맵과 자연어 분석 리포트를 생성합니다.
